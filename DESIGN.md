# AI-SAST：智能静态应用安全测试平台

## 项目简介

AI-SAST 是一个基于大语言模型的智能静态应用安全测试平台，通过多Agent协作的方式，实现代码漏洞的自动检测、分析、修复建议和持续优化。

**核心亮点**：
- 🧠 **AI Native 编排**：LangGraph StateGraph 驱动，LLM 自主决策扫描策略（而非写死流水线）
- 🔍 **七引擎漏洞检测**：Semgrep + Bandit + Gitleaks + Trivy + AI 深度分析 + 文件遍历 + 代码读取
- 🤖 **双层 Agent 架构**：编排层（LangGraph 宏观调度）+ 分析层（三 Agent 博弈微观分析）
- 📊 可视化管理界面
- 🧬 自我进化机制（从历史检测中学习）
- 💰 低成本运行（利用阿里云免费额度）

## 二、系统架构设计

### 2.1 架构理念：AI Native 双层 Agent 架构

**关键设计原则**：
- **编排层（宏观）**：LangGraph StateGraph + LLM 自主决策「扫什么、用什么工具、扫几轮」
- **分析层（微观）**：三 Agent 博弈系统深度分析「这个漏洞是不是真的、多严重、怎么修」

```
传统SAST:     代码 → 规则引擎 → 结果
AI Enhanced:  代码 → 固定流水线(semgrep→bandit→ai) → 结果
AI Native:    代码 → LLM编排Agent(自主决策) → 动态调用工具 → 三Agent博弈分析 → 结果
```

### 2.3 核心理念对比

| 方面 | 传统SAST | AI Enhanced | 本项目(AI Native) |
|------|----------|-------------|-------------------|
| **核心** | 规则引擎 | LLM 辅助分析 | LLM 自主编排 + 多 Agent 博弈 |
| **决策方式** | 预定义规则匹配 | 固定流水线 + AI 补刀 | LLM 动态决策 + 工具按需调用 |
| **工具角色** | 规则是全部 | 工具为主，AI 为辅 | 工具是 Agent 的「手脚」，LLM 是「大脑」 |
| **灵活性** | 低，需人工更新规则 | 中，流水线固定 | 高，LLM 根据代码结构自适应 |
| **误报处理** | 难以自适应 | 单 Agent 判断 | 三 Agent 博弈 + 仲裁收敛 |

### 2.4 双层 Agent 架构详解

```
编排层（LangGraph）── 管「宏观调度」
├── 决定先扫什么、后扫什么
├── 决定用哪些工具组合
├── 决定要不要对某文件深挖
└── 控制成本和轮次

分析层（三 Agent 博弈）── 管「微观分析」
├── 审查 Agent：这个发现是不是真的？
├── 安全 Agent：CVSS 多少？攻击路径是什么？
├── 修复 Agent：怎么修？成本多大？
└── 仲裁器：三方达成共识 → 输出最终结论
```

### 2.5 为什么这样设计？

1. **AI 是核心**：LLM 具备代码理解能力，可以处理规则无法覆盖的复杂场景
2. **工具是辅助**：Semgrep/Bandit/Gitleaks/Trivy 提供快速、确定性的检测，减少 AI 调用次数（省钱）
3. **双层解耦**：编排层负责「做什么」，分析层负责「怎么做」，各自独立演进
4. **可扩展**：未来可以给编排层添加更多工具（如 CodeQL、DAST 扫描器），给分析层添加更多 Agent 角色
5. **符合实际**：人类安全团队也是「项目经理调度 + 专家深度分析」的工作模式

### 3.2 博弈讨论机制

```python
# Agent协作伪代码
class AgentOrchestrator:
    def analyze(self, code):
        # 第一轮：各自独立分析
        agent1_result = review_agent.analyze(code)
        security_agent_result = security_agent.analyze(code, agent1_result)
        fix_result = fix_agent.analyze(code, security_agent_result)
        
        # 多轮博弈讨论
        for round in range(MAX_ROUNDS):
            # Agent间互相质疑
            challenges = self.exchange_challenges(
                agent1_result, 
                security_agent_result, 
                fix_result
            )
            
            # 根据质疑调整观点
            if self.has_convergence(challenges):
                break
                
            agent1_result = review_agent.revise(agent1_result, challenges)
            security_agent_result = security_agent.revise(security_agent_result, challenges)
            fix_result = fix_agent.revise(fix_result, challenges)
        
        # 最终仲裁
        return self.arbitrate(agent1_result, security_agent_result, fix_result)
```

### 3.3 Token优化策略

| 策略 | 说明 | 节省比例 |
|------|------|----------|
| 分层调用 | 只有可疑代码才调用AI | ~60% |
| 缓存机制 | 相似代码复用结果 | ~30% |
| Prompt压缩 | 精简输入，去除冗余 | ~20% |
| 异步批处理 | 合并多个小请求 | ~15% |

## 四、技术选型

### 4.1 技术栈

| 层次 | 技术 | 理由 |
|------|------|------|
| **前端** | Vue3 + Vite + Element Plus | 轻量、快速、组件丰富 |
| **后端** | FastAPI (Python) | 异步高性能、类型安全 |
| **数据库** | SQLite + Redis | 轻量级，适合2G内存 |
| **向量库** | ChromaDB | 轻量、嵌入式、无需额外服务 |
| **AI 编排** | LangGraph + LangChain | StateGraph 状态机编排，支持动态决策、断点续跑、可视化 |
| **SAST引擎** | Semgrep + Bandit + Gitleaks + Trivy | 开源、多语言、覆盖代码漏洞+密钥泄露+依赖CVE |
| **AI模型** | 通义千问-Turbo | 国内访问快、免费额度充足 |
| **容器化** | Docker + Docker Compose | 一键部署、环境一致 |

### 4.2 为什么选择这些技术？

**SQLite而非PostgreSQL**：
- 2G内存限制，SQLite更轻量
- 单文件部署，备份简单
- 性能足够（SAST不是高并发场景）

**ChromaDB而非Milvus**：
- 嵌入式，无需额外服务
- 内存占用小
- Python原生支持

**Semgrep而非CodeQL**：
- 安装简单，无需复杂配置
- 规则YAML编写，学习成本低
- GitHub原生支持

## 十、核心技术点详解（简历用）

### 10.1 项目技术亮点

#### 1. 多Agent协作系统
- **技术点**：基于LLM的多Agent协作、博弈论应用
- **难点**：Agent间如何高效通信、如何达成共识
- **解决方案**：设计"辩论-仲裁"机制，通过结构化消息传递和置信度评分实现收敛

#### 2. AI增强的静态分析
- **技术点**：LLM代码理解、漏洞模式识别
- **难点**：如何在保持低误报率的同时提高检出率
- **解决方案**：采用"Semgrep初筛 + AI复核"的分层策略，Semgrep过滤明显非漏洞，AI专注复杂场景

#### 3. Token优化
- **技术点**：Prompt工程、缓存策略、批处理
- **难点**：如何在有限预算下完成大量扫描
- **解决方案**：实现多级缓存（代码级、文件级、项目级）+ 智能Prompt压缩

#### 4. 向量知识库
- **技术点**：Embedding、语义检索、RAG
- **难点**：如何高效检索相关漏洞模式
- **解决方案**：基于ChromaDB构建漏洞知识库，支持相似漏洞检索

#### 5. 自我进化机制
- **技术点**：在线学习、反馈循环、规则优化
- **难点**：如何从有限反馈中持续改进
- **解决方案**：设计反馈标注系统 + 自动规则生成器

### 10.2 AI技术覆盖

| 技术领域 | 本项目应用 | 掌握程度要求 |
|----------|------------|--------------|
| **LLM应用** | 通义千问API调用、Prompt工程 | ⭐⭐⭐⭐ |
| **Agent系统** | 多Agent协作、角色定义、博弈机制 | ⭐⭐⭐⭐ |
| **RAG** | 向量知识库、语义检索 | ⭐⭐⭐ |
| **Embedding** | 代码向量化、相似度计算 | ⭐⭐⭐ |
| **NLP** | 代码理解、漏洞模式识别 | ⭐⭐⭐ |
| **MLOps** | 模型部署、性能监控 | ⭐⭐ |
| **数据工程** | 数据管道、特征工程 | ⭐⭐⭐ |

### 10.3 难点与解决方案

| 难点 | 解决方案 | 技术原理 |
|------|----------|----------|
| Agent协作效率低 | 异步并行 + 消息队列 | asyncio + Redis Queue |
| Token消耗过大 | 分层调用 + 缓存 | 语义哈希 + LRU Cache |
| 误报率高 | 多Agent投票 + 人工反馈 | 集成学习 + 主动学习 |
| 知识库检索慢 | 向量索引 + 分层检索 | HNSW + 两级索引 |
| 实时性要求高 | WebSocket + 增量更新 | 全双工通信 |
